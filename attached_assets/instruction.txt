# O2O Master Refactor Prompt for Replit

## Files to attach

Attach **the full current project zip** of the O2O codebase. Use the archive that contains the existing monorepo with these folders:

* `artifacts/o2o`
* `artifacts/api-server`
* `artifacts/admin-panel`
* `lib/db`
* any shared packages under `lib/`
* `package.json`, `pnpm-workspace.yaml`, lockfiles, and all current source files

If the file picker allows only one file, attach **the project zip**. Do not attach a trimmed export, screenshots, or a partial folder.

---

## Master prompt

You are working inside an existing O2O monorepo. Your job is to **inspect the current codebase, verify what is already implemented, fix what is broken, improve what is incomplete, and deliver a production-ready result**.

This is **not a rewrite**. Do **not** rebuild the project from scratch.

### Non-negotiable goals

* Keep the **existing UI/design** unless a bug forces a minimal fix.
* Work only with the **existing tech stack**.
* Do **not** add new unrelated technologies.
* Do **not** introduce Django, Python, AWS, Redis, or Expo.
* Use **React Native CLI**, not Expo.
* Remove any Expo-specific code, imports, config, scripts, or dependencies.
* The app must work **perfectly on Android and Web**.
* Remove or delete all iOS-specific code paths and files if they are present.
* Do not write any iOS code.
* Preserve current features, but make them actually work.
* No mock data in production screens.
* No placeholder handlers.
* No fake UI flows.
* No half-connected APIs.

### Current stack to keep

* **Frontend mobile/web**: React Native CLI + React Native Web
* **Backend**: Node.js + Express.js + TypeScript
* **Database**: PostgreSQL
* **ORM / schema**: Drizzle ORM + drizzle-kit
* **Validation**: Zod / drizzle-zod
* **Realtime**: Socket.IO
* **Auth**: JWT
* **File upload**: Multer and current upload flow already in the repo
* **Logging / security**: existing pino / helmet / cors / rate limit stack already in the repo
* **Monorepo tooling**: pnpm workspaces, strict TypeScript

### What to do first

1. Read the full repository structure.
2. Inspect all existing screens, API clients, shared libraries, backend routes, DB schema, socket handlers, upload logic, and auth logic.
3. Identify what is already implemented, what is partially implemented, and what is broken.
4. Fix bugs in the current architecture before adding anything new.
5. Only create new files when the existing codebase has no correct place for the fix.
6. Prefer improving existing files over adding duplicates.
7. Make sure every change works in the current architecture and does not break Android or Web.

### Core instruction

For **every screen, backend route, DB model, socket event, upload flow, and auth flow**:

* verify whether it is implemented
* verify whether it works correctly
* if implemented incorrectly, fix it properly
* if missing, implement it using the existing architecture
* if partly implemented, finish it cleanly
* if duplicated, remove the duplicate and keep the real implementation
* if mock data exists, replace it with real API-backed data

## What to verify and fix across the app

### 1) Frontend screens

* check if it already exists
* inspect whether the UI is functional or still placeholder code
* confirm all buttons, forms, mutations, queries, and navigation routes work
* replace mock data with real backend data
* ensure loading states, error states, and empty states exist where needed

* recent chats
* recent groups
* recent channels
* unread counts
* search users/friends/groups/channels
* notifications and friend requests entry points

### 4) Friend system

Verify and implement:

* search users by username
* send friend request
* accept/reject request
* notification creation

* correct realtime updates

### 5) Notifications

Verify and implement a real notification system for:

* friend request
* friend accepted
* friend rejected
* new message
* group invitation
* channel updates
* bid events
* offer events
* order events

### 6) Private chat

Verify and implement the existing chat system so it supports:

* text

* image
* video
* file
* audio
* poll
* location
* reply
* edit
* delete

### 7) Groups

### 11) Profile and settings

* notifications settings

### 12) Database integration

Inspect the PostgreSQL schema and Drizzle setup.
Make sure:

* tables are correct
* relations are correct
* indexes are correct
* unique constraints are correct
* cascade behavior is correct
* soft delete behavior is correct where needed

### 13) Socket.IO

Inspect and fix Socket.IO event flow so events are actually emitted and received where needed.
Verify existing events and handlers, and implement missing ones using the current architecture.

### 14) Storage and uploads

Inspect and fix file upload flow so it works end-to-end for:

* images
* videos
* documents
* avatars

* pagination
* infinite scrolling

* database query efficiency

### 16) Security

Verify and enforce:
